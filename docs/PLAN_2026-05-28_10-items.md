### Item 2 — Auto-updater 404 on `latest.yml` for v3.0.2

- **Files:** `src/main/updater-service.ts:120-131`,
  `electron-builder.yml:152-157`, GitHub release artifacts for v3.0.2
  and v3.2.0.
- **Root cause:** error came from an installed v3.0.2 build that
  doesn't have `latest.yml` published on that GitHub release.  Newer
  releases (v3.2.0) ship `latest.yml`; v3.0.2 doesn't, so the older
  binary's updater 404s.  Two-pronged fix.
- **Fix:**
  1. Attach `latest.yml` (and `latest-mac.yml` / `latest-linux.yml`)
     to the v3.0.2 + v3.2.0 GitHub releases retroactively — verify
     the CI run actually uploaded them.  (One-shot release maintenance.)
  2. Add a try/catch in `UpdaterService` that demotes 404s on
     `latest.yml` to a single console line instead of a stack trace,
     and surfaces a friendly "no updates available" status to the
     renderer.  Keep real network errors visible.

### Item 3 — API Models tab: only Claude shown + Copy Command broken

- **Files:** `src/renderer/components/models/ModelsPanel.tsx:484-491`
  (handleCopyCommand), `:1219-1221` (API tab render),
  `src/main/model-catalog-seed.ts:26-244` (seed entries).
- **Root cause:**
  - Catalog has Claude×2, Gemini, Aider, OpenRouter API entries but
    no standalone OpenAI / Gemini-direct / OpenRouter-direct entries.
    Aider entry says `provider: 'OpenAI'` but it's labeled "Aider" so
    the user reads the API tab as "only Claude is here."
  - `handleCopyCommand` calls `navigator.clipboard.writeText()` async

    silently failing, route to `ApiKeyModal` for that provider and
    chain into launch on save.
  - Copy command: await the clipboard write; show a 2s "Copied!" inline
    confirmation under the button; on rejection, show the command in a
    selectable text box as a fallback.

### Item 6 — LMM "+ New cycle" + focus-aware multi-tab

- **Files:** `src/renderer/components/lmm/LMMPanel.tsx:104-122`
  (handleNewCycle), `src/main/index.ts:1174-1202` (IPC handlers),
  `src/main/lmm-service.ts` (createCycle implementation),

- **Root cause:**

### Item 7 — Pop-out window + chat-skin divergence

- **Files:** `src/main/index.ts:875-945` (popout creation),
  `src/renderer/components/models/PopoutView.tsx:20-38`,
  `src/renderer/components/models/EmbeddedTerminal.tsx:138-155`,
  `src/renderer/components/chat-skin/ChatSkinOverlay.tsx:119-153`.
- **Root cause:**
  - "paneId p_root not found" comes from EmbeddedTerminal's 1.5s probe
    against `models.listRunning()` — if the main window's PTY died or
    the popout subscribed after a transient drop, the probe fires.
    PopoutView doesn't pass `profile`, so the popout mounts ChatSkin
    with `profile=undefined`.
  - The garbled "Mulling…" repeat in the main window vs clean chat in
    the popout = two ChatSkinOverlay instances independently parsing
    the same `onData` stream.  Their parsers diverge when one mounts
    after some bytes have already been consumed.
- **Fix:**
  - Store popout metadata (paneId, profile, label) in the
    `popoutWindows` map on main; expose `models.getPopoutMeta(paneId)`
    IPC; PopoutView fetches it before mounting EmbeddedTerminal.
  - Replace the `paneId not found` cold message with a "Re-attaching…"
    spinner + 5s retry; only show the fatal message if the PTY really
    is gone (check `models.listRunning()` once more after the retry).
  - Chat-skin sync: stop persisting toggle in `localStorage` per pane;
    move state to App.tsx (`chatSkinByPane: Record<string, boolean>`)
    and pass down via props.  Main + popout instances of the same
    paneId always read the same source of truth.
  - Parser convergence: route raw bytes through a single canonical
    parser per paneId living in main-process state, expose parsed
    events via IPC.  Each ChatSkinOverlay subscribes to parsed
    events (idempotent, ordered) instead of raw bytes.  This is
    the only durable fix for the "two parsers diverge" class of bug.
    *(If single-parser refactor is too much for v3.2.1, ship a
    cheaper fix first: have the popout subscribe to a replay of
    parsed events stored in a ring buffer on main, then catch up.)*

## Phase B — v4.0.0 (HF integration + rebrand, ~1-2 weeks)

### Item 5 — Hugging Face integration

**Scope (user-confirmed v1):** @huggingface/hub SDK + Ollama bridge.
No Transformers.js in v1.

#### Architecture

- **Main process:** `src/main/huggingface-service.ts` wraps
  `@huggingface/hub`.  Exposes IPC for:
  - `hf.search({query, task, sort, limit, gguf?})` — returns model
    cards (id, downloads, likes, tags, files, license).
  - `hf.modelInfo(repoId)` — full card + GGUF file list with quant
    variants.
  - `hf.download(repoId, filename, {dest})` — streams to user-data
    `hf-cache/` with progress events.
  - `hf.importToOllama(repoId, quant)` — either:
    (a) spawn `ollama run hf.co/<repoId>:<quant>` (preferred,
        Ollama pulls), or
    (b) download GGUF locally, write Modelfile, `ollama create`.
    Pick (a) by default, fall back to (b) if user opts in.
  - `hf.listCached()` / `hf.removeCached(repoId)` — manage local
    cache, surface size in Settings.
- **Renderer:** new `HFPanel` mounted as a sibling tab to
  ModelsPanel (or as a sub-tab inside Models).  Three sub-tabs:
  **Browse**, **Cached**, **Research**.
- **Search UX (Browse):** full-text input + filters (Task,
  Library, License, Min downloads, GGUF only).  Result cards: name,
  org, last updated, downloads, likes, quant variants if GGUF.
  "Import to Ollama" button → progress bar → toast on completion.
- **Cached:** lists locally downloaded HF files with size + remove.
- **Research section (uncensored / experimental):**
  - **Disabled by default.**  Settings → Advanced → "Enable
    Research Catalogs" toggle with a disclaimer:
    > Research catalogs surface community-curated model lists,
    > including uncensored / experimental models that may generate
    > harmful content.  Models are not reviewed by Hugging Face or
    > by this app.  Use only in sandboxed environments.  You assume
    > all responsibility for use.
  - When enabled, exposes a curated allowlist of HF Collections
    (start with `chwoo/uncensored-models`, expand from user input).
  - Models from the Research section run in an isolated Ollama
    profile (separate models directory via `OLLAMA_MODELS` env
    override) so they don't mix with general use.  Audit-log every
    prompt to a local file the user can review.

#### Right panel resize + wider default

- ModelsPanel + HFPanel: bump default width from current to ~420px.
- Add a resize handle on the left edge of the right panel using
  `react-resizable-panels` (already in deps).  Persist width in
  user-data.

#### Hugging Face GitHub repos worth borrowing patterns from

Reference: https://github.com/huggingface, https://huggingface.co/models

- `huggingface/transformers` — model card schema + tokenizer loading.
- `huggingface/datasets` — streaming download + caching layer.
- `huggingface/optimum` — quantization picker UI patterns.
- `huggingface/huggingface.js` — source of `@huggingface/hub`; copy
  their cache directory + OAuth patterns directly.
- `ggerganov/llama.cpp` (community, but key) — GGUF naming
  conventions; reference for quant tag display.
- Surface these as references in `journal/HF_INTEGRATION_DESIGN.md`
  written alongside the implementation.

### Item 9 — App rename

**Brainstorm (3 finalists for user pick, presented in v4.0.0 PR):**

1. **Forge** — professional + memorable; metal-workshop metaphor
   suits "AI model smithing"; echoes GitHub Forge.  Risk: name

   craft.  Stands out from tech-jargon names; appeals to researchers.
   Risk: harder to type / pronounce in some markets.
3. **Catalyst** — agentic, synthesis-focused; chemistry metaphor of
   "code + models = catalyst for build/ship."  Risk: AWS already
