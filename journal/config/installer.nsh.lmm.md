# LMM: build/installer.nsh

> File: `build/installer.nsh` · LOC: ~310 · Role: NSIS bootstrap script — downloads Node + Claude CLI at install time; (Cat 8) optionally downloads + installs Ollama based on a wizard prompt.

## Phase 1: RAW

This file owns the install-time bootstrap. v1.1 designed the original "download Node + npm-install Claude CLI" flow (documented in `INSTALLER_REDESIGN.lmm.md`). Cat 8 adds two things:

1. **Opt-in Ollama install.** Beta.1 bundled it unconditionally — surprise 2 GB download, no progress, no warning. Rolled back to detection-only. Cat 8 restores the install path BUT only after a MessageBox at the start of `customInstall` asks the user. Default is No (so silent `/S` installs on CI / kiosk don't trigger the 2 GB pull).

2. **Wizard chrome.** `electron-builder.yml` flips `oneClick: true → false` and wires the new BMP assets (header, sidebar, uninstaller sidebar — generated by `build/gen-installer-assets.mjs`). This turns the silent one-click installer into a real Wise-Setup-style wizard with Back/Next buttons, where the Ollama MessageBox fits naturally between the welcome page and the install progress.

The MessageBox approach (vs a custom nsDialogs page) is a deliberate trade-off:
- **Pro**: portable across electron-builder versions. The custom-page macro names (`!macro customWelcomePage`, `!macro preInit`, etc.) vary between builder releases. MessageBox is plain NSIS — works regardless.
- **Pro**: simpler. ~10 lines of NSIS instead of ~50 lines for a custom page + page-insertion hook.
- **Con**: visually a step backwards from the wizard chrome. The MessageBox is OS-styled, not custom-painted.
- **Con**: can't include rich layout (radio buttons, links, etc.) — just buttons and text.

Net call: the user explicitly named the goal as "modernize the look + Ollama opt-in" — the look is handled by the wizard mode + BMPs; the opt-in is handled by MessageBox. Future iteration can swap the MessageBox for an nsDialogs page if it bothers anyone.

## Phase 2: NODES

### Node 1: `Var OllamaWantsInstall` — string "0"/"1"
NSIS doesn't have booleans; we use strings. Why it matters: clear semantics, no implicit type coercion.

### Node 2: Step 0 runs BEFORE the bootstrap steps
The Ollama question is asked at the start of `customInstall`, BEFORE Node download. Why it matters: gives the user a chance to opt out without first sitting through Node + npm install.

### Node 3: `/SD IDNO` default on the MessageBox
Silent installs (`/S` flag) get No by default. Why it matters: kiosk / CI installs shouldn't surprise-pull 2 GB.

### Node 4: Step 5 branches on `$OllamaWantsInstall` + presence
Three cases: already present (skip), not present + opted in (download + install), not present + opted out (skip with log). Why it matters: belt-and-suspenders — never install twice over an existing Ollama.

### Node 5: Soft-fail on both download and installer-exit
A bad Ollama download or non-zero installer exit shows a MessageBox + continues with Studio install. Why it matters: third-party tool failures should never block Studio.

### Node 6: `/SILENT /NORESTART` flags to Ollama installer
Inno Setup conventions (which Ollama's installer is). `/SILENT` keeps the Ollama install UI invisible during our wizard so the user sees one progress flow. Why it matters: the user already consented to install; making them click through Ollama's wizard too would be friction.

### Node 7: `--max-time 1800` on the Ollama curl
30-minute total cap. Ollama is ~2 GB; on a slow connection that's not unreasonable. Why it matters: Node's `--max-time 300` (5 min) would falsely fail on real-world bandwidth.

## Phase 3: REFLECT

### Core insight
**This is a two-step trust ladder.** Step 0 asks for consent before the heavy work. Step 5 honors that consent by either pulling the bytes or politely declining. The bootstrap never silently installs a 2 GB dependency.

### Resolved tensions
- **Node 3 (default No) vs Node 2 (ask up front)**: a user paying attention will see the MessageBox and make a real choice. A user running `/S` (silent) gets the safe default. Both paths are reasonable.
- **Node 5 (soft-fail) vs the no-rollback semantics elsewhere**: hard failures (Node download / SHA) Abort the whole install. Ollama is a soft dependency — its failure shouldn't undo Studio. The distinction matches the user's actual mental model.

### Hidden assumptions
- Assumed: Ollama's installer URL (`https://ollama.com/download/OllamaSetup.exe`) is stable. Challenge: it has been for years, but if the URL pattern changes the download fails soft. Documented.
- Assumed: Ollama's installer accepts `/SILENT /NORESTART`. Challenge: it does as of the latest releases. If a future version drops them, we get the visible Ollama wizard popping up mid-Studio-install. Annoying but recoverable.

## Phase 4: SYNTHESIZE

### What this file should become
A stable two-stage bootstrap. The Cat 8 additions are the last big feature here. If we ever want a real nsDialogs page (more visual, more options), this file is where it lives — but the MessageBox is fine for the foreseeable.

### Actionable items
- [ ] Upgrade the MessageBox to an nsDialogs page when an electron-builder version with a stable `customWelcomePage` / similar macro is committed to.
- [ ] Add a real progress bar via nsisdl or NSISdl-DL for the Ollama download (curl + DetailPrint doesn't show byte progress).
- [ ] Verify the Ollama URL via a HEAD request before curl-ing — could add a "the install URL is unreachable, want to skip?" prompt.
- [ ] Test on a real fresh Windows VM with various AV setups; document any Defender quirks.

### Risks
- A 30-minute curl is a long time to be blocked on a single connection. Network blips during the download produce a soft failure — user gets a MessageBox saying "Ollama install failed; you can install later." That's the right behavior.
- Bundle size still small: Node 30 MB + Claude CLI ~few MB + (optional) Ollama 2 GB. Without Ollama opt-in, install is 30-40 MB. With opt-in, network during install determines duration.
